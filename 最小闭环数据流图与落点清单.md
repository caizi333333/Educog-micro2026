# 最小闭环数据流图与落点清单（功能视角）

> 目标：围绕你提的三件事——**学习进度/学习路径的无感稳定**、**仿真解释一致性（可验证证据链）**、**教师端可用性（班级-学生-薄弱点-布置任务闭环）**——把“现状数据流”画清楚，并给出“落点清单（该改哪些文件/接口）”。

---

## 0. 快速结论（现状 vs 缺口）

### 已经具备的“闭环骨架”
- **学习进度采集与保存**：`src/hooks/useTrackProgress.ts` → `/api/learning-progress`（含队列/熔断/超时/只增不减倾向）→ `prisma.learningProgress`
- **测评**：`/api/quiz/submit` 写 `quizAttempt`，并会“反向触发”一次 `/api/learning-progress` 更新章节完成条件
- **学习路径展示**：`/learning-path` + `src/app/learning-path/learning-path-client.tsx`，支持 URL 参数与 localStorage 恢复、fallback 计划
- **仿真**：已经有一个确定性的 8051 仿真器核心 `src/lib/simulator.ts`，并且 UI 已展示“执行追踪 + diff”（`ExecutionTrace`/`ControlPanel`）
- **教师端 Dashboard**：`/teacher` 页面 + `/api/teacher/dashboard` API（聚合学生测验、实验、学习进度）

### 当前最影响“产品粘性/真实课堂落地”的缺口
1) **学习进度“恢复”链路缺失**：进度能存，但页面刷新/换设备后无法“无感恢复到上次进度”（目前 `LearningModuleWithProgress` 只展示本次会话计算/保存返回值）。
2) **学习路径“持久化与版本化”缺失**：数据库里有 `LearningPath` 与 `/api/learning-path/save`，但 `LearningPathClient` 目前并没有把生成的 plan 落库，也没有“版本化/可更新”的策略。
3) **教师端数据契约不稳定**：Dashboard 的 `chapterMastery` 键名依赖 `chapterId` 的格式，但前端做了 `ch1/ch01/chapter-1` 多路猜测；如果实际入库是别的格式，会导致大量字段为 0，从而触发“演示数据”回退。
4) **布置任务/作业闭环缺失**：教师端 UI 有“推送学习任务/布置课前测试”等入口，但缺少对应的数据模型与 API（目前是按钮占位）。

## 1) 学习进度 → 保存 → 恢复 → 推荐（现有数据流）

### 1.1 数据流图（现状）

```mermaid
flowchart LR
  A[学习页面内容] --> B[LearningModuleWithProgress]
  B --> C[useTrackProgress]
  C -->|POST| D[/api/learning-progress]
  D --> E[(Prisma: LearningProgress)]
  D --> F[(Prisma: UserPointsTransaction/User.totalPoints)]

  G[测评页面 QuizClient] -->|POST| H[/api/quiz/submit]
  H --> I[(Prisma: QuizAttempt)]
  H -->|触发一次 POST| D

  H --> J[localStorage: assessment-results]
  J --> K[/learning-path]
  K --> L[LearningPathClient]
  L --> M[generateLearningPlan(可选)]
  L --> N[generateFallbackPlan(必有)]
```

### 1.2 关键落点（你现在已有的“做对了”的地方）
- **只增不减倾向**：
  - 前端：`useTrackProgress.ts` 内部 `progressRef.current = Math.max(old, new)`
  - 后端：`/api/learning-progress` 会用 `calculateCompletionPercentage` 重新计算，并写回 progress（建议继续强化“max(old, new)”语义，见后文）
- **并发/队列/超时/熔断**：`useTrackProgress.ts` 已经在朝“稳定保存”演进（这部分方向很正确）
- **测评反向影响完成度**：`/api/quiz/submit` 在章节测验场景会再调用一次 `/api/learning-progress`，把 quiz 完成条件纳入完成判定（闭环意识很好）

### 1.3 现状痛点（为何还做不到“无感恢复”）
**恢复链路缺失**：目前你有：
- `/api/learning-progress` 的 GET（单查/列表）
- `/api/learning-progress/batch` 的 POST（批量查）

但在学习页组件层：
- `LearningModuleWithProgress` **没有在 mount 时去拉取已有进度并回填**到 `useTrackProgress` 的内部 state/ref
- `useTrackProgress` 本身也没有 `initialProgress/initialTimeSpent` 的注入入口

因此“刷新页面/换设备”后会出现：
- UI 进度条回到 0 → 只有重新滚动/停留后才会慢慢回升
- 学习体验不连续，影响粘性（这是你最关心的点）

### 1.4 建议的最小改造（让进度真正无感稳定）

#### A) 给 `useTrackProgress` 增加“初始化注入点”
**建议新增 options**：
- `initialProgress?: number`
- `initialTimeSpentSeconds?: number`
- `initialLastAccessAt?: string`

并提供一个方法：
- `hydrateFromServer(progressRecord)`：把服务端记录同步到 `progressRef/currentProgress/totalTimeSpent` 等关键状态

落点：`src/hooks/useTrackProgress.ts`

#### B) `LearningModuleWithProgress` 在 mount 时加载已有进度（单章）
新增 effect：
- 调 `/api/learning-progress?moduleId=...&chapterId=...`（或 batch API）
- 成功后调用 `hydrateFromServer`

落点：`src/components/LearningModuleWithProgress.tsx`

#### C) 服务端明确“只增不减”的最终裁决
在 `/api/learning-progress` 的 update 分支里：
- `progress = Math.max(existing.progress, actualCompletion)`
- `timeSpent = existing.timeSpent + validatedTimeSpent`（已是累加）
- `completedAt` 只在首次完成时写入（已是）

落点：`src/app/api/learning-progress/route.ts`

## 2) 仿真 → 可验证视图 → 解释一致性（现有数据流）

### 2.1 数据流图（现状）

```mermaid
flowchart LR
  A[SimulationPage] --> B[useSimulator]
  B --> C[Simulator: run / stepWithTrace]
  C --> D[src/lib/simulator.ts]
  D --> E[SimulatorState]
  D --> F[ExecutionTraceEntry(step/pc/changes)]

  E --> G[StatusMonitor / MemoryViewer]
  F --> H[ExecutionTrace(追踪视图)]
  F --> I[ControlPanel(控制台/变化摘要)]
```

### 2.2 关键发现：你已经有“可验证证据链”的核心能力

这点非常关键：**你当前的仿真并不是“纯 AI 猜结果”**，而是已经有一个可重复的确定性内核：
- 仿真核心：`src/lib/simulator.ts`
  - 有 `SimulatorState`（寄存器/端口/内存/外设）
  - 有 `ExecutionTraceEntry`（step/pc/instruction + reg/mem/port/flag diff）
- UI 已经在使用证据链：
  - `ExecutionTrace`：展示逐步 trace 与每步 diff（`src/components/simulation/ExecutionTrace.tsx`）
  - `ControlPanel`：把最后一步 diff 摘要输出到“控制台”（`src/components/simulation/ControlPanel.tsx`）

换句话说：**“可验证视图”你已经有 70%**；剩下的 30% 是把“教学解释”也绑定到这个确定性证据链上，形成“一致性解释”。

### 2.3 当前“解释一致性”的主要缺口
1) 缺少一个统一的“解释模型”（Explain Model）：
   - 目前 UI 的解释更多是状态/变化的展示（是什么），而非教学解释（为什么）。
2) 错误定位缺少“可引用证据”：
   - `fault` 是字符串（来自 run/step 的异常信息），但没有结构化信息（错误类型/定位行/相关寄存器状态/修复建议）。
3) AI 的角色没有“收口”到 trace：
   - 你有 AI flows（例如 `ai-study-assistant`、`code-simulation-flow`），但仿真页当前主要使用的是确定性 Simulator；如果后续要引入 AI 辅助解释，建议**让 AI 解释 trace**，而不是让 AI 生成 state。

### 2.4 建议的最小改造：新增“解释层”而不动仿真内核

```mermaid
flowchart LR
  A[Simulator.stepWithTrace] --> B[ExecutionTraceEntry]
  B --> C[ExplainTrace(新)]
  C --> D[UI: 解释面板/错因定位/修复建议]

  subgraph ExplainTrace(新)
    E[规则解释器(确定性)] --> C
    F[AI 教学解释(可选)] --> C
  end
```

#### A) 先做“规则解释器”（确定性、可测试、可控）
给每个 trace entry 做一个 `explain(entry, prevState?)`：
- 输出结构建议：
  - `summary`：一句话解释本步做了什么
  - `why`：为什么会有这些变化（例如 MOV 写端口锁存器、DJNZ 影响 PC 跳转）
  - `evidence`：引用本步 diff（寄存器/端口/内存变化）作为证据
  - `commonMistake`：结合课程常见错误（例如直接/立即数混淆、端口读改写问题）

落点建议：
- 新增：`src/lib/simulator-explain.ts`（纯函数、方便单测）
- 复用现有知识库：`src/lib/problem-graph.ts`（里面已经有“常见错误/解决方案”结构，适合做错因映射）

#### B) 再做“AI 教学解释”（可选、但要强约束）
让 AI 输入：`traceEntry + 前后状态摘要 + 规则解释器产物`，输出：
- 更自然的中文讲解
- 更贴合课堂的类比

落点建议：
- 新增 flow：`src/ai/flows/simulator-explain-flow.ts`（输出必须是结构化 schema）
- UI 端只展示“结构化字段”，避免 AI 胡编影响一致性

## 3) 教师端：班级/学生/薄弱点/任务（现有数据流）

### 3.1 数据流图（现状）

```mermaid
flowchart LR
  A[/teacher] --> B[TeacherPage]
  B -->|GET| C[/api/teacher/dashboard]
  C --> D[(Prisma: User(role=STUDENT))]
  C --> E[(Prisma: QuizAttempt)]
  C --> F[(Prisma: UserExperiment)]
  C --> G[(Prisma: LearningProgress)]
  C --> H[(Prisma: UserActivity 24h)]
```

### 3.2 你现在“已经够用”的部分
- `/api/teacher/dashboard` 的聚合思路是对的：
  - 总览：总学生数/今日活跃/平均测验分/实验完成率
  - 学生表：avgQuizScore + chapterMastery + 实验完成数
  - 预警：avgQuizScore < 60
- `/teacher` UI 已经能展示：
  - 知识掌握度热力图（按章节）
  - 实验完成柱状图
  - 预警学生

### 3.3 当前主要缺口（为什么难进真实课堂）
1) **缺少“班级实体”与“选课关系”**
   - 目前是直接抓所有 `role=STUDENT` 用户，真实课堂通常至少要按班级/课程筛选。
   - `User.class` 是 string，但没有班级表/教师授课班级关系。
2) **缺少“作业/任务/测验布置”的数据模型与 API**
   - `/teacher` 页面有“推送学习任务/布置课前测试”等按钮，但没有后端闭环。
3) **薄弱点/章节掌握的“ID 契约”不稳定**
   - Dashboard 前端用 `ch1/ch01/chapter-1` 多路猜测 `chapterId`，如果学习进度落库用的是别的命名（如 `chapter-1` 或 `item-1`），热力图会全 0 → 直接回退到 mock 数据。

### 3.4 建议的最小闭环（教师端 v1）
只做能跑通课堂最小流程：
- 教师选择班级 → 看到班级总览与薄弱点
- 教师发布任务（章节/实验/测验）→ 学生完成后自动汇总完成率

建议新增最小数据模型（概念级，未写入代码）：
- `Class`（班级）
- `Enrollment`（学生-班级关系）
- `Assignment`（任务/作业，包含截止时间、发布人）
- `AssignmentItem`（任务条目：章节/实验/测验）
- `AssignmentSubmission`（学生完成/提交记录）

## 5) 建议的执行顺序（按“最小闭环 + 最大增益”）
1) **进度恢复**（最直接提升粘性）：学习页 mount 拉进度 + hook hydrate + 服务端 max 裁决  
2) **学习路径落库**（让推荐可持续）：生成后保存到 LearningPath（并能从 DB 恢复）  
3) **仿真解释层（规则解释器）**：先把“为什么”可重复地解释出来，再考虑 AI 润色  
4) **教师端班级化**：先能筛班级，再谈布置任务  
5) **任务布置 v1**：最小 Assignment 闭环（章节/测评/实验 + 截止 + 完成率）
